Strategies for Minimizing Database Vulnerability


This week at DBC served as my introduction to databases and gave me my first taste of SQL, the most common database language currently in use. While the topic may seem dry at first, it quickly becomes apparent that all of the shiniest bells and whistles that we love in web apps are entirely dependent on databases.


Databases store data about, well... EVERYTHING. Tweets. User accounts. Payment information. Grades. Likes. Sites visited. Exchange rates. Recent transactions. Availability of resources. Contacts. The list goes on (along with several cross referenced lists linked by foreign key ids).


BUT - with the powerful access that apps provide to data, they also open the door for users to try to manipulate this data in unsafe ways.


One of the most pervasive attacks on web apps is SQL injection. Dynamic web apps are built to allow users to interact with databases. Users send their information, or requests for information from the front end interface. The app then embeds this user input into a SQL query - translating user text into a structure that the Database Management System (DBMS) can understand. A hacker can manipulate this translation and insert an additional SQL query into the input field. This additional command, or injection, can wreak havoc by accessing privileged information or damaging the database records.


UNLESS the proper precautions are taken.


Luckily for new devs like me, there are many excellent resources that allow us to safeguard our apps from the scourge of SQL Injection.

The Open Web Application Security Project, or OWASP, provides a SQL Injection Prevention Cheat Sheet. Very generous, no? They write that to prevent SQL Injection, "developers need to prevent user supplied input which contains malicious SQL from affecting the logic of the executed query."


Here is brief description of best practices - though for specific code and strategy, check the above links.

  1. Prepared Statements: the app inserts user input fields into variables that are plugged into SQL templates called 'Prepared Statements'. The input is escaped, and cannot contain viable SQL code.
  2. Stored Procedures: basically the approach as above, but rather than the templates being stored within the app, the templates are stored and accessed within the Database itself.
  3. Escape All User Input: not reccomended as a practice, though may be necessary as the most cost effective security measure for legacy products.


Additionally, Devs should build the following considerations into app design.

  • Least Privilege: consider the level of access each type of user should have to the DBMS. And ensure that they have the bare minimum access necessary to query the data relevant for their needs. No more, and no unintentionally broad access.
  • White List Input Validation: Assemble a list of valid characters for each input field - and only allow users to type values from this 'white list' of valid inputs.
